# MyTDD: 테스트 주도 개발 워크플로우

이 스킬이 활성화되면, 프로덕션 코드를 작성하기 전에 반드시 Red-Green-Review-Refactor 사이클을 따른다.

## 진행 절차

0. **사전 준비 (테스트 실행 환경 확인)**
   - 작업을 시작하기 전에 프로젝트에 테스트 프레임워크와 빌드 도구가 이미 설정되어 있는지 확인한다.
   - 설정되어 있지 않다면, 사용 가능한 빌드 도구(예: MSBuild, cl, cmake, dotnet 등)를 먼저 찾는다.
   - 표준 테스트 프레임워크 도입이 여의치 않다면(외부 의존성 설치가 불가능한 경우 등) 자체 제작한 최소한의 테스트 하네스를 만들어 사용할 수 있다. 이 경우 어떤 도구/하네스를 선택했는지와 그 이유를 사용자에게 알린다.

1. **Red (실패하는 테스트 작성)**
   - 요구사항을 가장 작은 단위로 쪼갠다.
   - 아직 구현되지 않은 동작을 검증하는 테스트를 먼저 작성한다.
   - 테스트를 실행해 실제로 실패하는지(그리고 기대한 이유로 실패하는지) 확인한다. 컴파일 에러로 인한 실패와 assertion 실패를 혼동하지 않는다.
   - 테스트를 실행했는데 기존 구현이 이미 일반화되어 있어 통과하는 경우, 억지로 프로덕션 코드를 변경하지 않는다. Green 단계를 생략하고 바로 Review로 넘어가되, 왜 추가 구현이 필요하지 않았는지를 리포트에 명시한다.

2. **Green (테스트를 통과시키는 최소한의 코드 작성)**
   - 테스트를 통과시키는 데 필요한 최소한의 프로덕션 코드만 작성한다.
   - 아직 요구되지 않은 기능이나 일반화를 미리 추가하지 않는다.
   - 테스트를 실행해 통과하는지 확인한다.

3. **Review (정리 및 승인 요청)**
   - 코드를 임의로 리팩토링하지 않는다 — 이 단계는 오직 보고와 승인 요청만 수행한다.
   - Red/Green 단계에서 수행한 내용을 정리한 리포트를 작성해 사용자에게 제시한다. 리포트에는 다음을 포함한다:
     - 추가한 테스트와 그 의도
     - Green 단계에서 작성한 최소 구현 내용
     - 발견된 중복, 어색한 이름, 구조적 문제 등 리팩토링이 필요한 지점
     - 각 지점에 대해 어떻게 리팩토링할 것인지에 대한 구체적인 계획
   - 리포트 제시 후 사용자에게 리팩토링 진행 여부와 계획에 대한 승인을 받는다.
   - 사용자가 계획을 수정하라고 하면 리포트를 갱신하고 다시 승인을 받는다. 승인받기 전에는 4단계로 넘어가지 않는다.

4. **Refactor (승인된 계획 실행)**
   - Review 단계에서 승인받은 계획에 따라서만 구조를 개선한다. 승인되지 않은 변경은 추가하지 않는다.
   - 테스트가 통과하는 상태를 유지하면서 중복 제거, 이름 개선, 구조 정리를 수행한다.
   - 매 리팩토링 단계마다 테스트를 다시 실행해 여전히 통과하는지 확인한다.
   - 새로운 동작을 추가하지 않는다 — 리팩토링은 동작 변경이 아니다.

5. 위 사이클을 요구사항이 모두 구현될 때까지 반복한다.

## 규칙

- 대응하는 실패하는 테스트 없이 프로덕션 코드를 먼저 작성하지 않는다.
- 한 번에 하나의 테스트/동작만 다룬다. 여러 테스트를 한꺼번에 작성한 뒤 구현하지 않는다.
  - 예외: 사용자가 명시적으로 여러 개의 유사한 테스트 케이스(예: 여러 엣지 케이스)를 한 번에 요청한 경우, 각 테스트가 기존 프로덕션 코드로 이미 통과한다면 모두 추가한 뒤 한 번에 실행 결과를 보고할 수 있다. 다만 그중 하나라도 프로덕션 코드 변경이 필요하다면, 그 시점부터는 각 테스트를 최소 단위(Red-Green)로 나누어 처리한다.
- 각 단계(Red/Green/Refactor)마다 실제로 테스트를 실행하고 결과를 확인한다 — 실행하지 않고 통과를 가정하지 않는다.
- 테스트는 이 프로젝트에 이미 설정된 테스트 프레임워크와 실행 방식을 따른다. 테스트 실행 명령을 모르면 먼저 확인한다.
- Review 단계의 리포트 제시와 승인 요청 없이 Refactor 단계로 진행하지 않는다.
- 각 사이클(Red → Green → Review → Refactor)이 끝날 때마다 어떤 테스트를 추가했고 어떤 최소 구현을 했으며 어떤 리팩토링을 수행했는지 한두 문장으로 사용자에게 알린다.
